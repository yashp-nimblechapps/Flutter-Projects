8) .family and .autoDispose methods 

i)family
Problem without .family
Suppose you want greetings for different users:

    User 1 → “Hello Yash”
    User 2 → “Hello Admin”
    User 3 → “Hello Guest”

Without .family, you’d need multiple providers:

    greetingUser1Provider
    greetingUser2Provider
    greetingUser3Provider

Solution:  
    final greetingAsyncProvider =
    AsyncNotifierProvider.family<GreetingAsyncNotifier, String, int>(
        () => GreetingAsyncNotifier(),
    );

Now the provider accepts a parameter (int id).

Usage:
    ref.watch(greetingAsyncProvider(1));
    ref.watch(greetingAsyncProvider(2));


ii) autoDispose

Problem without .autoDispose
Riverpod keeps providers alive forever by default.

That means:
    open screen → API call
    leave screen → provider still in memory 
    come back → old cached data
    Sometimes this is bad.

.autoDispose = automatic cleanup
AsyncNotifierProvider.autoDispose.family(...)

Now:
    Screen opened → provider created
    Screen removed → provider destroyed

Memory freed

////ProviderObserver 

(It is used to monitor providers)
    It watches:
        When provider is created
        When provider value changes
        When provider is destroyed

    How to Use
        void main() {
            runApp(ProviderScope(
                observers: [MyObserver()],
                child: MyApp(),
                ),
            );
        }
    Then create:
        class MyObserver extends ProviderObserver {


@override. // Provider value changes, Provider state changed (It gives: old value, new value)
didUpdateProvider(  
    ProviderBase provider,
    Object? previousValue,
    Object? newValue,
    ProviderContainer container
) {}
    
@override. // Provider is created for first time, A provider started
didAddProvider(
    ProviderBase provider,
    Object? value,
    ProviderContainer container
) {}

@override  // Provider is destroyed, Provider ended 
(Example: autoDispose provider removed, Screen closed)
didDisposeProvider(
    ProviderBase provider,
    ProviderContainer container
) {}

//Mostly ProviderObserver is use for?
Debugging, Logging, Tracking state changes, Finding rebuild issues

Quick Recap :

StateProvider → for simple state (counter, toggle)
StateNotifierProvider → for complex logic & multiple methods
FutureProvider → simple one-time async
AsyncNotifier → async + custom methods + more control
.family allows a provider to accept parameters and create multiple instances based on input
.select() reduces unnecessary rebuilds by listening to only a specific part of state
